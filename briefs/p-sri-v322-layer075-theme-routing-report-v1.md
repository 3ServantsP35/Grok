## 1. Purpose

This document is the first actual draft of the **Layer 0.75 Theme Routing and Ingest Report** for the v3.2.2 architecture.

The purpose of the report is to make Layer 0.75 legible to users by reporting:
- which top-level branches are active
- what each active branch currently says
- how much confidence to place in each branch read
- how aggressively each branch should be deployed
- what route-specific warnings or contradictions matter
- how macro and phase context are conditioning each branch
- what evidence supports the routing and deployment calls

Like the Layer 0 and Layer 0.5 reports, this report should stay **pure in authority** while becoming highly useful in implication.

That means it should:
- report routing state accurately
- preserve branch ownership boundaries
- surface contradictions instead of flattening them into a fake single answer
- explain what Layer 0.75 is telling the lower layers about sleeve deployment

It should **not**:
- replace Layer 0 macro authority
- replace Layer 0.5 phase authority
- replace branch-native internal logic with SRI prose
- decide final portfolio weights by itself unless a later architecture version explicitly promotes it into that role

## 2. Core reporting framework

The Layer 0.75 report should be organized around eight durable reporting fields.

1. **Routing state**
2. **Active branch set**
3. **Branch state**
4. **Branch confidence**
5. **Deployment intensity**
6. **Preferred exposure style**
7. **Warnings / contradictions**
8. **Evidence package**

These fields reflect the architecture definition in `SRI-Engine-Tutorial-v3.md` and the downstream doctrine in the APE ingest and Visser grounding briefs.

### 2.1 What Layer 0.75 owns

Layer 0.75 has three jobs:

1. **Routing**
   - determine which top-level branches are active
2. **Ingest**
   - preserve branch-native outputs instead of flattening them
3. **Deployment conditioning**
   - decide how aggressively active sleeves should be deployed given Layers 0 and 0.5 plus route-native intelligence

This means Layer 0.75 is not just “theme commentary.”
It is the layer that turns macro context plus branch intelligence into deployable sleeve posture.

### 2.2 Branch set in v3.2.2

The current top-level branches are:
- **MSTR**
- **Visser / APE**
- **All-Weather**

The report should treat each branch as a real sleeve with its own ownership boundary, state, and deployment logic.

### 2.3 Confidence-factor framework

Confidence at Layer 0.75 should answer:

> How clear is the current branch-level read and deployment instruction?

A simple v1 scale is enough:
- **High confidence** = branch state, macro conditioning, and deployment posture align cleanly
- **Moderate confidence** = the branch read is usable, but contradictions or missing inputs remain
- **Low confidence** = branch state is stale, conflicted, weakly evidenced, or too dependent on unresolved overlays

Layer 0.75 confidence should be influenced by:
- freshness and quality of the branch-native input
- coherence with Layer 0 and Layer 0.5
- presence of unresolved contradictions
- whether timing depends on a human overlay not yet supplied

### 2.4 Deployment-intensity framework

The report should classify deployment intensity clearly.

A simple v1 ladder:
- **Increase**
- **Maintain**
- **Selective / staged**
- **Reduce**
- **Defer / pause**

This is the most actionable output of Layer 0.75.

## 5. Routing-state block

## 5.1 Core routing rule

The routing rule should be published explicitly in every version of this report:

- **0% branch allocation** → branch is moot
- **>0% branch allocation** → branch is active

This matters because Layer 0.75 should not imply a branch is “off” when it is simply unallocated by mandate.

## 5.2 Current routing interpretation

Current routing should be interpreted as:
- **allocation-driven activation first**
- **deployment conditioning second**

That means Layer 0.75 decides:
- whether a branch matters for the current owner
- how much trust and urgency to apply to that branch now

It does **not** decide whether the owner conceptually believes in a branch at all. That is a mandate and allocation question.

## 5.3 Why this matters

This is the cleanest way to separate:
- strategic sleeve existence
from
- tactical sleeve deployment

That distinction is core to the v3.2.2 architecture.

## 7. Branch-confidence block

## 7.1 What confidence should mean here

Branch confidence at Layer 0.75 is not just about branch conviction.
It is about whether the branch can be **used operationally right now**.

That means confidence should be influenced by:
- branch-native conviction
- freshness of the ingest artifact
- coherence with Layer 0 and Layer 0.5
- availability of needed timing overlays
- severity of unresolved contradictions

## 7.2 Current doctrine-based confidence view

The current architecture implies a rough maturity ranking:
- **MSTR:** highest operational maturity
- **Visser / APE:** high strategic value, but dependent on ingest freshness and advisory boundaries
- **All-Weather:** conceptually stable, but dependent on clearer sleeve rules as the broader service/reporting stack matures

That means the report should be willing to distinguish:
- **high branch conviction, moderate deployment confidence**
- **fresh thesis, stale artifact**
- **good macro alignment, weak timing clarity**

This is exactly where Layer 0.75 becomes useful.

## 8. Deployment-intensity block

## 8.1 What the report should decide

For each active branch, the report should classify deployment intensity as one of:
- **Increase**
- **Maintain**
- **Selective / staged**
- **Reduce**
- **Defer / pause**

## 8.2 Why deployment intensity is the key output

Layer 0.75 is where branch intelligence becomes a usable sleeve instruction.

The user does not only need to know whether MSTR or Visser is strategically attractive.
They need to know:
- should this sleeve be deployed now?
- should it be staged?
- should it be capped despite conviction?
- should the sleeve wait for lower-layer confirmation?

That is why deployment intensity should be the centerpiece of the report.

## 8.3 Most likely deployment philosophy

My current base-case interpretation is:
- Layer 0.75 should **rarely** produce all-or-nothing branch verdicts
- it should more often express branch views through **deployment intensity, exposure style, and warning logic**

That means contradictions should usually resolve as:
- stage, don’t force
- reduce, don’t erase
- defer, don’t pretend conviction vanished

That feels fully aligned with the v3.2.2 architecture.

## 9. Preferred exposure-style block

## 9.1 What this field should mean

For each branch, the report should say what style of exposure is preferred under current conditions.

Examples:
- shares over leverage
- leverage acceptable but staged
- reserve-biased sleeve expression
- narrow expression vs broad basket expression
- human-timing overlay required before offense increases

## 9.2 Why this matters

This is where Layer 0.75 becomes more precise than abstract deployment posture.

A branch may remain attractive, but the preferred expression may change because:
- macro is improving but still fragile
- Howell remains defensive
- branch conviction is strategic, but timing is poor
- concentration risk is elevated

This field should therefore become one of the most educational parts of the report.

---

## 10. Warnings / contradictions block

## 10.1 Why this block is essential

The architecture explicitly says Layer 0.75 should not collapse disagreements into a false single answer.

This means the report should publish contradictions like:
- strong thesis, weak timing
- good timing, weak thesis
- macro headwind, branch conviction intact
- branch conviction high, benchmark posture still cautious
- owner override vs engine recommendation

## 10.2 Most likely interpretation philosophy

My current view is that Layer 0.75 should become the architecture’s main **contradiction interpreter**.

That means when different layers disagree, Layer 0.75 should usually say:
- what is conflicting
- which layer owns what
- how the conflict should change deployment intensity
- what would resolve the conflict

That is much more valuable than pretending the system has a perfectly unified answer when it does not.

## 11. Evidence package

The report should not stop at branch summaries. It should show the evidence behind them.

### 11.1 Required evidence categories

For each active branch, the evidence package should include as much of the following as is available:

1. **Routing evidence**
   - owner sleeve allocation / active status

2. **Macro conditioning evidence**
   - latest Layer 0 conclusion
   - latest Layer 0.5 conclusion

3. **Branch-native evidence**
   - MSTR branch internals, APE digest state, All-Weather state, etc.

4. **Freshness / health evidence**
   - generated time
   - freshness state
   - stale / aging warnings

5. **Contradiction evidence**
   - specific mismatch between branch-native view and SRI overlays

6. **Deployment-evidence note**
   - what evidence is driving the deployment intensity call

### 11.2 MSTR evidence pattern

The MSTR evidence package should eventually point to:
- branch-native force / force-field outputs
- MSTR-specific state / deployment read
- any route-level caution from macro or Howell overlays

### 11.3 Visser evidence pattern

The Visser evidence package should eventually point to:
- `mstr-knowledge/ape_intel.md`
- digest freshness state
- era / thesis / route posture
- routed insights and strategic warnings
- explicit note that APE is advisory intelligence, not benchmark authority

### 11.4 All-Weather evidence pattern

The All-Weather evidence package should eventually point to:
- reserve / ballast logic
- branch-specific durability rationale
- macro and Howell reasons for stronger or lighter ballast

### 11.5 Reporting principle

If a branch lacks enough evidence, the report should say so directly.

That means Layer 0.75 should be allowed to publish states like:
- strategically valid, operationally under-evidenced
- conviction present, freshness weak
- deployment deferred pending overlay

That honesty is better than overclaiming precision.
